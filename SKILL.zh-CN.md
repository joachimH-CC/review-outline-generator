# 复习提纲生成器

本文件是 `SKILL.md` 的中文对照版，方便中文母语者阅读。真正用于技能触发的入口仍然是 `SKILL.md`。

## 定位

这个 skill 用于根据用户提供的本地课程资料，生成结构化的 Markdown 复习产物。它是一个“AI 工作流 skill”，不是一个“全自动程序”：

- 脚本负责从课件中提取文本和图片
- AI 负责结合资料、约束和模板组织最终 Markdown
- 除非用户明确允许联网补充，否则内容必须以本地资料为依据

## 适用范围

适合：

- 某门课程的复习提纲
- 基于课件、讲义、习题整理期末复习资料
- 分章节的 Markdown 复习笔记
- 在没有习题时，仅根据课件整理章节重点
- 在没有习题时，根据用户提供的章节序号和重点，从教材 PDF 整理章节复习提纲

不适合：

- 没有资料的泛化知识讲解
- 没有本地文件支撑的“总结整门课”
- 自动评分、自动覆盖率统计、逐句来源审计

## 所需输入

至少需要用户提供本地课程资料：

- 课件或阅读材料：`PPT/PPTX`、`PDF`、`DOCX`

强烈建议同时提供：

- 课后习题文件，如果用户想要“按题目整理”的输出
- 明确的考试范围或章节范围
- 没有课后习题、需要从教材 PDF 整理时，提供具体章节序号和复习重点
- 参考格式文件，如果用户希望输出贴近某个既有 Markdown 样式

如果请求太宽泛，不要直接编造正式提纲。要么要求补资料，要么走下面定义的降级路径。

## 支持的输入模式

### 模式 A：说明文件驱动

如果用户提供了基于 `assets/general-template.md` 填写的说明文件，就把它当作主要控制面。

从中读取：

- 科目名称
- 章节范围
- 题目来源
- 章节与题号映射
- 章节与复习重点映射
- 图表偏好
- 是否允许联网补充
- 输出目录

也接受自然语言说明文件，只要其中仍然清楚指向了：

- 考点文件
- 课件目录
- 习题目录
- 教材 PDF；当无习题路径基于教材而非课件时必须提供
- 章节重点；当 `question_source: "keypoints"` 由用户给定重点驱动时必须提供
- 可选的参考格式文件
- 章节范围或最终输出要求

在正式生成前，先做预检：

- 检查链接到的文件和目录是否存在
- 检查参考样例是否存在；如果不存在，要明确回退到默认模板
- 检查题号映射是否超出提取出的题目范围
- 记录 warning，而不是静默猜测

### 模式 B1：对话式，且资料与范围齐全

如果用户没有提供说明文件，但明确给出了：

- 本地课程资料
- 章节范围或考试范围

那么先提取资料，再按与模式 A 相同的逻辑整理输出。

### 模式 B2：对话式，有资料但没有范围

如果用户有资料，但没有明确考试范围：

- 先检查资料
- 如果允许联网，可选地生成一份范围/模板草案
- 所有推断项都标记为 `[联网推断，请确认]`
- 展示给用户，等待确认后再生成正式提纲

联网推断只用于起草范围、权重或模板草案，不能在未确认前直接混入正式正文。

### 模式 B3：资料不足

如果用户没有提供可用的课程资料：

- 不生成正式提纲
- 输出缺失项清单
- 明确说明下一步需要哪些文件或字段

## 正式支持的降级路径

只支持以下两条：

1. 有课件或教材 PDF，没有习题

- 设置或建议 `question_source: "keypoints"`
- 生成按章节组织的重点概念、定义、图表说明和来源摘录
- 如果用户提供了具体章节序号和重点，使用 `chapter_focus_map`，并以教材 PDF 作为主要资料来源
- 不要把结果伪装成“按题目整理的习题提纲”

2. 有课件，有习题，但没有考试范围

- 先起草一份说明文件或范围模板
- 如果用户允许联网，可用联网推断补一版待确认草案
- 只有在用户确认范围后，才生成正式提纲

不把“只有习题、没有课件”当作正式支持路径。

## `question_source` 的含义

保留原有字段名，但要明确三者不是同级自动化能力：

- `exercises`
  主路径。适用于有习题和题号映射的情况。输出以题目或题组为组织单位。

- `keypoints`
  降级路径。适用于没有习题、但有课件或教材 PDF 的情况。输出以章节重点、用户给定重点、定义、表格、图示说明为主。

- `custom`
  受限路径。仅当用户在”自定义说明”中明确给出输出结构时使用，且结果仍然必须以提供的资料为依据。

## 分析题章节的动态识别

当考点文件或说明文件中同时包含问答映射（问答）和分析题关键字（分析题）时，skill 必须区分两者：

- 考点文件中列在”问答”下的章节 → 以问答格式输出，使用 `question_source=exercises`
- 考点文件中列在”分析题”下的章节 → 以分析题格式输出，使用四段式详细结构（见组装规则）
- 同时在两个列表中出现的章节 → 同时包含问答和分析题两个部分

当仅提供 `chapter_focus_map`（keypoints 模式）时，所有章节均使用重点概念格式。

当 `analysis_keywords` 存在但某章节没有单独的问答映射时，在该章节中包含分析题部分。

## `reference_format` 的定位

把 `reference_format` 当作“样式参考”，不要当成“结构化驱动文件”。

可以参考它的：

- 标题层级
- 分组方式
- 标注风格
- 图片摆放
- 表格语气

不要承诺：

- 自动抽出严格 schema
- 完全复刻排版

## 提取流程

在没有现成原生提取能力时，使用附带脚本：

- `scripts/extract_pptx.py <pptx_dir> <text_dir> <image_dir>`
- `scripts/extract_docx.py <docx_dir> <text_dir> <image_dir>`
- `scripts/extract_pdf.py <pdf_file_or_dir> <text_dir> <image_dir>`
- `scripts/build_review_outline.py <spec_path> [--output-dir DIR] [--work-dir DIR] [--refresh]`

只有当结构化提取不够用、且确实需要从屏幕手工截取时，才使用 `scripts/screenshot.py` 作为 Windows 手工兜底。

`build_review_outline.py` 是优先推荐的稳定执行路径。它会解析说明文件、执行预检、运行提取、选择题目、格式化 PPT 补充，并写出最终 Markdown。

其余提取脚本只负责提供原始文本和图片资产，不负责完整组装。

如果教材 PDF 是扫描版、没有可抽取文本，只能把页面图片作为兜底素材。不要假装已经语义读取教材；正式生成前应要求 OCR 结果、页码范围，或取得用户同意后走人工/视觉辅助提取。

## 组装规则

### 通用规则

组织输出时：

- 答案和笔记必须以提取出的资料为依据
- 课件是主要支撑材料
- 当用户在没有课后习题时提供章节序号和重点，教材 PDF 是主要支撑材料
- 如果输出按题目组织，就把习题作为主线
- 匹配不强时，要明确说明，不要假装确定
- 有帮助但不直接回答题目的课件内容，放入引用块作为补充
- 使用教材 PDF 摘录时，要保留可读结构，并标记为 `教材整理`
- 如果 slide 里有 `（1）（2）...`、`A、B、C、...` 等分点结构，必须拆成多行，不能压成一整段

如果某个点在提供资料中找不到：

- 保持该段简短
- 明确标注，例如 `[PPT中未找到对应内容]`
- 不要编造缺失内容

### 标题层级规范

使用统一的标题层级：

- H1：章节标题，格式 `# 第X章 {章节名称}`
- H4：题型分组标签，格式 `#### 问答` / `#### 重点概念` / `#### 分析题`
- H5：问答题号，格式 `##### X. {完整题干原文}` — 不要加"问题"等前缀，直接使用原始题号
- H6：分析题子标题，格式 `###### 分析题 — {主题}`，或重点标题 `###### 重点{X}. {名称}`

### 内容保真度

保留原始习题文本结构：

- 保留课后习题原文的段落结构，不要过度改写或重组
- 使用原始措辞，仅去除明显的格式伪影（页码、docx 元数据）
- 保留原始编号如 `(1)`、`A.` 等
- 定义和 PPT 补充说明放入引用块（`>`）
- 目标是干净的阅读体验，不是完整重写

### PPT补充格式

使用统一的格式：

```
> **PPT补充（Slide {页码}）：**
> - {要点1}
> - {要点2}
```

当涉及多个 slide 时，每个 slide 单独一行 `**PPT补充（Slide X）：**`。

### 分析题章节结构（针对分析题章节）

当一个章节被识别为分析题章节（如用例图、顺序图、类图、数据建模等），使用详细的四段式结构：

```
###### 分析题 — {主题}

**一、{基本概念}**
{解释该建模概念的目的和背景}

**二、{核心元素}**
{逐一列出并解释每个核心元素}

**三、{进阶/关系}**
{解释元素间的关系、复杂概念、约定规范等}

**四、{分析题示例}**
{结合课后习题的题目，逐步讲解示例}
```

四段式结构是分析题章节的输出主体，不要使用标准问答格式（`#### 问答` + `##### X.`）。

## 图表处理

把 `diagram_mode` 当成”输出偏好”，但在复习提纲场景下要再加一条硬规则：

- 只要题目明确要求 `用例图`、`顺序图`、`活动图`、`状态机图`、`数据流图`、`类图`、`实体关系模型`、`组织结构图`、`角色权限矩阵` 等可视化内容，就默认直接插入源图片
- 只有当源图片缺失或定位冲突时，才允许把是否继续升级为需要确认的问题

**重要：图片相关性规则** — 仅插入确实相关且能识别的图片。如果提取出的图片无法辨认（空白、部分裁剪、无意义装饰、渲染异常等），跳过它，不要为了”有图”而插入无意义的图片。

然后再应用模式本身的行为：

- `export`
  默认模式。优先使用原资料里已有的图片，必要时补少量说明。

- `mermaid`
  当内容适合抽象成结构图时，可输出 Mermaid 源码块，但不承诺渲染。

- `plantuml`
  当内容适合抽象成结构图时，可输出 PlantUML 源码块，但不承诺渲染。

如果图表是 AI 辅助生成而非原资料导出，必须标注 `[AI生成，仅供参考]`。

## 联网补充

`web_enrich: true` 仅表示用户允许做有限的解释性补充。

规则：

- 课程内容仍以本地资料为主
- 联网补充必须标记为 `[联网补充]`
- 不能用联网内容偷偷替代缺失的课程资料
- 必须区分“联网推断范围”和“联网补充说明”

前者用于在范围缺失时起草模板；后者用于在主结构已落地后补充说明。

## 输出文件

典型输出包括：

- 每章或每组章节一个 Markdown 文件
- 当 `generate_overview` 开启时，生成 `复习提纲总览.md`
- 当启用图片输出时，生成 `images/` 目录

如果没有自定义参考格式，就以 `output-examples/通用参考格式.md` 作为默认结构参考。本课程工作区的默认参考格式基于 `D:\文档\课程作业资料\软件需求分析\00\第一章.md`。

## 轻量校验

只做轻量校验，不承诺自动覆盖率统计。

检查：

- 预期输出文件是否生成
- 图片路径和明显的内部链接是否失效
- 联网推断、联网补充、AI 图表是否被正确标注
- 是否存在“只有标题没有正文”的空壳章节
- keypoints 模式章节是否包含用户给定重点，或明确选取出的章节概念
- 显式要求图示的题目，在源图片存在时是否真的插入了图片
- PPT 补充块是否把长枚举错误地压成了一整行
- 题号映射是否越界

如果发现明显缺口，就按以下类别报告：

- 缺失章节
- 依据较弱的章节
- 需要用户确认的疑问点

## 约束

始终遵守：

1. 不编造课程内容。
2. 不把推断范围当成已确认范围。
3. 不把联网内容静默混入原始资料答案。
4. 不把 `keypoints` 输出伪装成 `exercises` 输出。
5. 需求不明确时，先索要缺失文件，或先确认草案范围；仅基于教材 PDF 生成时，缺章节序号或重点就是阻塞项。
6. 不要把 `$grill-me` 用在日常的图片插入或格式处理上；只有真正阻塞生成时才问。
